QUANTIZING IN SHADER CODE

The floor is handled in this line of the shader code:

	case 50: ALBEDO = diffuse*game_floor_color*simple_hash_noise(surface_position, 1.0); ALPHA=1.0; break;

It's currently calling the surface_position raw. What if it called the quantized surface position instead?

Let's add a quantizing formula to the top of the shader, for quantizing vec3's:

	vec3 quantized_vec3(vec3 p) {
		return floor(p*100.0)/100.0;
	}

As you can see, it's a lot easier to write the function in shader code--it was built around these kinds of shenanigans.

Now, instead of just calling for the surface position in the floor section of the switch logic, call for the quantized surface position instead:

	case 50: ALBEDO = diffuse*game_floor_color*simple_hash_noise(quantized_vec3(surface_position), 1.0); 
	ALPHA=1.0; break;

Run the scene.

THIS SOLVES TWO PROBLEMS, NOT JUST ONE 

You'll notice that two problems get solved. Our original artifact, the chaotic aura around the moving spheres, has gone away.

An additional problem gets nuked as well: The floor's pattern remains fixed even as the camera moves around the scene. Before, the camera's own movement was introducing small changes into the hash noise function that were getting picked up.

The pattern still seems to shift too much when more than a few meters away. Try tweaking the quantization function to be in tenths instead:

	return floor(p*10.0)/10.0;

This causes a pixellated result. But it also causes the result at the far end of the platform to be steadier.

I'm partial to Worley noise, partly because I'm a fan of the look, and also because it relies on distance functions. Since raymarching is all about the distance function, this is probably the smallest leap.

In the very beginning of this work, there was a shader of "Hello, World" level complexity. It looked like this:

	col = vec3(length(uv));

When the UV origin was mapped to zero, it produced a black blob in the middle (where the length of the vector was zero) that greyed out to white the further you were from the center. Each pixel color was its measurement of distance to the origin. Worley noise works similarly, except instead of measuring distance to one point, it measures the distance to 9 surrounding points in 2D space, and 27 points in 3D space (essentially turning a Sudoku puzzle into a Rubik's cube).

You might choose to follow the tutorial in the Book of Shaders, but since there's some Godot-specific challenges, I'm going to translate some elements of their work into a Godot package.

Create a new scene, create a 3D node called WorleyBasics, create a MeshInstance3D with a quad mesh, override the surface material in the Inspector, and give it a new shader called worley_basics.gdshader. Start with this foundation:

shader_type spatial;
render_mode unshaded;

void fragment() {
	vec2 uv = UV;

	uv = vec2(uv.x,(1.0-uv.y));
	
	uv = uv*2.0 - 1.0 ;
	
	vec3 col = vec3(length(uv));
	col = clamp(col, 0.0, 1.0);
	col = pow(col, vec3(2.2));
	
	ALBEDO = col; ALPHA = 1.0;
}

As of now, making the color equal the length of the UV effectively measures the length to the center, since the UV origin is at the middle of the quad, and what we're treating as 0,0.

If you wanted to make a point at 0.25, -0.5, how would you make the pixel effectively measure the distance to that point? The "uv" vec2 is a vec2, and that new point is also a vec2. Start by declaring that new vec2. Call it "point1".

void fragment() {
	//Add point 1, an arbitrary point in the bottom rightish.
	vec2 point1 = vec2(0.25, -0.5);

Now, you need to know the distance between wherever you are in uv space and the point. There's an appropriately named built in for that, distance(). Distance accepts two floats, two vec2s, or two vec3s. We're in 2d space at the moment.

Update "col" to use this instead:

	col = vec3(distance(uv,point1));

Your shader is now measuring the distance to point1.

Worley noise creates patterns by finding the distance to the nearest point among several. So add a point, and try to think of a function that will deliver the nearest point, and return that distance. We've already done something similar with the big object list SDF. The function needs to ask, "for any given UV, and with two defined points, I need to examine the distance from the UV to each point and return the smallest distance".

I came up with this, which is, unsurprisingly, basically the same series of min() functions that we saw with the object list SDF:

float nearest_point_distance(vec2 uv, vec2 p1, vec2 p2){
	float d = distance(uv, p1);
	d = min(d, distance(uv,p2));
	return d;
}

Add another point with a location of your choosing. I went with:

vec2 point2 = vec2(-0.5,0.1);

The color line can get updated to instead return the results of the distance function that takes multiple points, going from this:

	col = vec3(distance(uv,point1));	

...to this:

	col = vec3(nearest_point_distance(uv,point1,point2));	


Try adding a third point. Note that your function will need to update to accomodate it.

	float nearest_point_distance(vec2 uv, vec2 p1, vec2 p2, vec2 p3){
		float d = distance(uv, p1);
		d = min(d, distance(uv,p2));
		d = min(d, distance(uv,p3));
		return d;
	}

And your color line will need an update:

	col = vec3(nearest_point_distance(uv,point1,point2,point3));	

Boom: We now have three points.

There's a lot of different directions we could go from here, and if you're already thinking of your own, you should experiment with that and see what happens.

But as you know, we expanded that function later on, to keep track of not just the distance, but a number to keep track of a material. You can just click over to your existing camera_gel.gdshader file to see it, but to save you the trouble, we added a min_n_mat function to replace min() and then updated the SDF list:

	vec2 min_n_mat(vec2 first_object_d_and_m, vec2 second_object_d_and_m){
		return
		(first_object_d_and_m.x < second_object_d_and_m.x)
		?
		first_object_d_and_m : second_object_d_and_m;
	}

	vec2 get_scene_distance(vec3 point) {
