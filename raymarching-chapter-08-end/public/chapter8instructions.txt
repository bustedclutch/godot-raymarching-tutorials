TRANSITIONING FROM GDSCRIPT TO SHADER CODE FOR QUANTIZING INPUTS
	
Before we embark on bigger journeys like clouds and procedural terrain generation, we need to figure out how to stop the floor from looking like some version of buggy orange TV static.

The last chapter introduced a quantizing function, which in gdscript, looked like this:

	func quantized_vector(vector : Vector3):
		var x = floor(vector.x*100)/100
		var y = floor(vector.y*100)/100
		var z = floor(vector.z*100)/100
		return Vector3(x,y,z)

I also tried simplifying the hash function to replace the big number at the end with a little one (literally the number one):

	func smoother_hash_noise(p: Vector3, t:float):
		return fractional_part(sin(  (p+Vector3(t,t,t)).dot(some_vector)  )*1)

QUANTIZING IN SHADER CODE

The floor is handled in this line of the shader code:

	case 50: ALBEDO = diffuse*game_floor_color*simple_hash_noise(surface_position, 1.0); ALPHA=1.0; break;

It's currently calling the surface_position raw. What if it called the quantized surface position instead?

Let's add a quantizing formula to the top of the shader, for quantizing vec3's:

	vec3 quantized_vec3(vec3 p) {
		return floor(p*100.0)/100.0;
	}

As you can see, it's easier to write the quantizing function in shader code as opposed to gdscript--it was built around these kinds of shenanigans.

Now, instead of just calling for the surface position in the floor section of the switch logic, call for the quantized surface position instead:

	case 50: ALBEDO = diffuse*game_floor_color*simple_hash_noise(quantized_vec3(surface_position), 1.0); 
	ALPHA=1.0; break;

Run the scene.

THIS SOLVES TWO PROBLEMS, NOT JUST ONE 

You'll notice that two problems get solved. Our original artifact, the chaotic aura around the moving spheres, has gone away.

An additional problem gets nuked as well: The floor's pattern remains fixed even as the camera moves around the scene. Before, the camera's own movement was introducing small changes into the hash noise function that were getting picked up.

The pattern still seems to shift too much when more than a few meters away, because at that distance, the outcome of the raymarch still has enough breadth that quantizing to the nearest hundredth isn't enough. Try tweaking the quantization function to be in tenths instead:

	return floor(p*10.0)/10.0;

This causes a pixelated result. But it also causes the result at the far end of the platform to be steadier. 

Switch back to hundredths.

I'm partial to Worley noise, partly because I'm a fan of the look, and also because it, too, relies on distance functions. Since raymarching is all about the distance function, this is probably the smallest pedagogical leap.

In the very beginning of this work, there was a shader of "Hello, World" level complexity. It had a distance function for a circle that looked like this:

	col = vec3(length(uv));

When the UV origin was mapped to zero, it produced a black blob in the middle (where the length of the vector was zero) that greyed out to white the further you were from the center. Each pixel color was its measurement of distance to the origin. (If you want to experiment with this function before going further, try turning it into a circle with a step function.)

Worley noise works similarly, except instead of measuring distance to one point, it measures the distance to 9 surrounding points in 2D space, and 27 points in 3D space (essentially turning a Sudoku puzzle into a Rubik's cube).

You might choose to follow the tutorial in the Book of Shaders, but since there's some Godot-specific challenges, I'm going to translate some elements of their work into a Godot package. I also want to break down the concept into an even simpler version.

Create a new scene, create a 3D node as the scene's root node called WorleyBasics, create a MeshInstance3D with a quad mesh, override its surface material in the Inspector, and give it a new shader called worley_basics.gdshader. Start with this foundation:

	shader_type spatial;
	render_mode unshaded;

As of now, making the color equal the length of the UV effectively measures the length to the center, since the UV origin is at the middle of the quad, and what we're treating as 0,0.

THE CLOSEST DISTANCE AMONG MULTIPLE POINTS: WORLEY NOISE

Worley noise creates patterns by finding the distance to the nearest point among several, and rendering a color based on the closest distance. 

Add a second point, and try to think of a function that will deliver the nearest point for any given UV location, and return that distance. We've already done something similar with the big object list SDF. The function needs to ask, "for any given UV, and with two defined points, I need to examine the distance from the UV to each point and return the smallest distance".

I went with the below code, which is, unsurprisingly, basically the same series of min() functions that we saw with the object list SDF:

float nearest_point_distance(vec2 uv, vec2 p1, vec2 p2){
	float d = distance(uv, p1);
	d = min(d, distance(uv,p2));
	return d;
}

Add another point with a location of your choosing. I went with:

vec2 point2 = vec2(-0.5,0.1);

The color line can get updated to instead return the results of the distance function that takes multiple points, going from this:

	col = vec3(distance(uv,point1));	

...to this:

	col = vec3(nearest_point_distance(uv,point1,point2));	

Hopefully, you now have a shader with two competing blobs.

Try adding a third point, and update the distance function to accomodate it.

For the voronoi basics shader, we upgraded worley basics from storing just the distance of the closest point to also packing that same point's arbitrarily named material id.

Instead, what if we stored both the distance to the closest point and the distance to the second closest point? If we followed the same pattern as before, we'd convert the distance float to a vec2, store the closest distance in the X component, and store the second closest distance in the Y component. And if that was all that you were doing, then it's likely how you'd solve this. 

But: Maybe we just like descriptive names. Or, more importantly, we want to be ready for situations where we might store five or six or seven components and we don't want to go crazy. 

If in some alternate universe vec3 didn't exist, and you could create your own data type with the keyword "magicvariablecreationword", then creating the vec3 would look something like this:

	magicvariablecreationword vec3 {
		float x;
		float y;
		float z;
	}

That magic variable creation word doesn't create an instance of that vec3 yet, it just describes the form that it takes, in particular, the name of each component and what variable type it is. 

That's essentially what "struct" does. So say we want to have a variable type that packs both the first and second closest distances, and we wanted a "double distance" variable type that stored two floats, we can declare it like this:

	struct dd {
		float first_closest_distance;
		float second_closest_distance;
	};

Just as we can create a vec3 with the name "camera_location" by writing this:

vec3 camera_location = vec3(0.0, 0.0, 3.0);

We are now empowered to create a variable that tracks both our distances. Pretend for just this moment that we wanted to declare a variable with this new "dd" type and store 100 as the first closest distance, and 50 as the second:

	dd distance_tracker = dd(100.0,50.0);

You still have to use what's called the "constructor" which is repeating "dd" before the parentheses.

Later on, if we wanted to change that 50.0 to a 75.0, we could do it with this verbiage:

	distance_tracker.second_closest_distance = 75.0;

struct dd {

Now, you need a new minimum function. It needs to say "hi, i'm keeping track of the smallest and second smallest distance, and if you give me a fresh distance, then I need to do one of three things:
	1. Do nothing, because the distance is greater than my currently stored distances
	2. Replace the second closest distance, if the new distance is closer than only the second closest distance 
	3. Replace the first closest distance and and move it to second place, in case the new distance is closer than both of my current distances 
	
Claude the chatbot's description of this process was to imagine an Olympics podium that only has room for first and second place, which I think is a good analogy. 

We'll call it "double_distance_keeper", and it'll return a dd type variable. Take a stab at coding it yourself. Here's my attempt, though the logical ordering might be problematic. We'll find out!

	dd double_distance_keeper(float fresh_distance, dd current_pair){
		if (fresh_distance > current_pair.first_closest_distance && 
		fresh_distance > current_pair.second_closest_distance){
			return current_pair;	
			}
		else if (fresh_distance < current_pair.second_closest_distance &&
		fresh_distance > current_pair.first_closest_distance){
			return dd(current_pair.first_closest_distance, fresh_distance);
		}
		else {
			return dd(fresh_distance, current_pair.first_closest_distance);
		}
	}

You might have noticed that my logic fails to properly deal with the possibility that the fresh_distance would exactly equal the distances in the current_pair. At present, the color line is now receiving a two component "dd" type on this line:

	col = vec3(nearest_point_distance(uv,point1,point2,point3));	

That line is a little confusing now that we've got a struct in play, so let's create a new variable to work from just above it:

CLOSING QUOTE FROM TEXTURING AND MODELING, A PROCEDURAL APPROACH, 3rd EDITION

Darwyn Peachey, in Chapter 2, had this to say:

	[I]t isn't hard to design a [white noise] function. [...] [W]hite noise isn't quite what is needed in a stochastic texture primitive. The repeatable pseudorandom function has an unlimited amount of detail, which is another way of saying that its values at adjacent points are completely independent of one another (uncorrelated). This sounds like what we want, but it proves to be troublesome because of the prevalence of point sampling in computer graphics. If we view an object from a new camera angle, the positions of the sample points at which the texture function is evaluated will change. A good PRN function will change its value markedly if the inputs change even slightly. Consequently, the texture will change when the camera is moved, and we don't want that to happen.

	Another way to look at this problem is in terms of aliasing. White noise has its energy spread equally over all frequencies, including frequencies much higher than the Nyquist frequency of the shading samples. The sampling rate can never be high enough to capture the details of the white noise.

	To keep our procedural textures stable and to keep them from aliasing, we need a stochastic function that is smoother than white noise. The solution is to use a low-pass-filtered version of white noise. (Low-pass filtered noise is sometimes called *pink noise*, but that term is more properly applied to a stochastic function with a 1/f power spectrum).

	The properties of an ideal noise function are as follows:
	-noise is a repeatable pseudorandom function of its inputs.
	-noise has a known range, namely, from -1 to 1.
	-noise is band-limited, with a maximum frequency of about 1.
	-noise doesn't exhibit obvious periodicities or regular patterns. Such pseudorandom functions are always periodic, but the period can be made very long and therefore the periodicity is not conspicuous.
	-noise is stationary--that is, its statistical character should be translationally invariant.
	-noise is isotropic--that is, its staticial character should be rotationally invariant.

I like this quote because it's a good review of both what we've experienced and where we still need to go. We already experienced the misery of the camera and object movement causing slight deviations in the noise, and the resulting chaos that introduced.

And even at the very small scale of our Godot scene, aliasing is already a serious problem--the closer parts of the scene have a Minecraft-pixel vibe whereas the edge of the platform reads as static. 

He then talks about a "low pass filter". As far as the GPU is concerned, it's unable to reach into the adjacent pixel to perceive what's happening there. But the noise function itself can still look different directions into its own reality.
