I ended the last chapter with a problem: The hash functions accept single numbers. And our hash function in particular only accepts unsigned integers.

Let's restate the issue as pseudocode that we can iterate on:

	Each input X and Y will have to:

		Individually get multiplied by a different odd, large constant, and
		Combined with each other via an XOR operation (the caret! ^), and
		Fed into the hash now that they are one binary number, and
		Split in two at the end, giving you two 16 bit numbers as output X and Y.

We're still in gdscript territory. We are still sticking, desperately, to abstracting out these functions as much as we can. The current hash function is just a bunch of code sitting under the ready() function, which honestly, I'm not bothered by.

But to prepare ourselves for rewriting this in shader code, let's keep the balance abstract.

We need a function, that takes two x and y values as inputs, and produces a single combined integer output that's usable for a hash function. That gives us something close to a header and a return:

	function vec2_to_uint_converter(vec2 input) -> integer

If you're paying attention, I've introduced a terrible bug already. More on that later, in the section about the terrible error.

We know x and y each need to be multiplied by a big number; I'll just write out x for now and plan to write y after I've figured x out:

	variable x = input.x 
	x *= big_hexadecimal_number

We know multiplying x by a big number exposes us to the possibility that x will become a very large number expressed in more than 32 bits, because the CPU will do math all the way up to 64 bits by default. We don't want that, so we'll mask it. We know that "F" means 1111 in hexadecimal, and that an AND operation on a 32 bit number that's all 1's will functionally drop a bigger number down to 32 bits. This keeps our GPU math and CPU math from diverging.

	x &= 0xFFFFFFFF

Finally, we know that we can combine x and y using an additional XOR operation. 

	variable result = x ^ y 

This gives us some not-quite-right-looking code to clean up and make pretty. 

	function vec2_to_uint_converter(vec2 input) -> integer
		variable x = input.x 
		x *= big_hexadecimal_number
		x &= 0xFFFFFFFF
		variable result = x ^ y

...a few beautification steps first:

	func vec2_to_uint(xy : Vector2) -> int: #This line is broken; we'll find out why later
		var x = xy.x
		var y = xy.y
		x *= 0x27d4eb2d #multiply by a big random odd constant
		y *= 0x165667b1 #multiply by some other big random odd constant

IT'S HERE THAT IT DAWNS ON ME THAT I'VE MADE A TERRIBLE MISTAKE 

All of this bitwise math is going to break under the hood.

If you haven't figured out why, ponder why! I never actually told you, so, don't feel bad if you have no idea.

WE SHOULD NEVER HAVE ACCEPTED A VEC2 AS THE INPUT 

Vector2's store their individual components as floats. Floats are not stored in the same way that integers are. In the binary number system, when you have floats, an arbitrary number of bits are reserved for what's called the "mantissa" and another part called the "exponent". Mantissa sounds like a cocktail you'd get at a bar or a campy name for a James Bond villain. So we'll stick with mantissa. We've already heard about the bit that's used to sign the integer, which is the far left bit called the "sign bit". Claude likens the exponent part to a magnifying knob that dials up the size of the mantissa. 

Sign---	Exponent---	Mantisssssssssssahhhhhhhhhhh!
0		0111 1111 	001 0000 0000 0000 0000 0000 

This will screw up our bitwise calculations for obvious reasons because these might as well be two different languages.

So: We should avoid contaminating our function with floats, as they'll introduce a completely different set of binary numbers, causing silent problems.

A refactoring (and renaming!) of the function is thus in order. Correct the first line, and the declaration of x and y:

	func combine_x_and_y_to_uint(x: int, y: int) -> int:
		x *= 0x27d4eb2d #multiply by a big random odd constant
		y *= 0x165667b1 #multiply by some other big random odd constant
		x &= 0xFFFFFFFF
		y &= 0xFFFFFFFF
		return x ^ y #return x combined with y using XOR

BUT WAIT: CAN'T THE SIGNED PORTION OF THE BIT SCREW EVERYTHING UP?

No. Not with the masking. Read below if you care about why.

On the GPU, we'll have access to a built-in function, uint(). It doesn't actually change the first bit from 1 to 0. It just causes the binary number to be interpreted differently. This binary number:

1111 1111 1111 1111 1111 1111 1111 1111  (also called 0xFFFFFFFF in hexadecimal)

...is interpreted as -1 normally if it's considered signed, but if you treat it as an unsigned integer, then it's interpreted as 4,294,967,295. The binary number stays the same.

We don't have a uint() in place here. So could that mess us up? Well, let's just take an example. Say I want to use x= -1. Using uint(x) on the GPU will relabel the binary number, but the binary number will still be that long string of ones.

But: What does -1 look like if you're in 64 bits? Well, I'll break it up into two lines, but it looks like this:

  hex:     0xFFFFFFFFFFFFFFFF

  first half:  1111 1111 1111 1111 1111 1111 1111 1111  
  second half: 1111 1111 1111 1111 1111 1111 1111 1111
  decimal (unsigned interpretation): 18,446,744,073,709,551,615   (= 2⁶⁴ − 1)
  decimal (signed interpretation):   −1

What about -3? In 64 bits?

  hex:     0xFFFFFFFFFFFFFFFD

  first half:  1111 1111 1111 1111 1111 1111 1111 1111 
  second half: 1111 1111 1111 1111 1111 1111 1111 1101

  unsigned interpretation:  18,446,744,073,709,551,613   (= 2⁶⁴ − 3)
  signed interpretation:   −3

So far, we're noticing a pattern: The first half is always a bunch of ones, and we chop those with a mask like we did in the hash function. That pattern holds past the 2.14 billion mark, which is when the GPU's integer length effectively stops it from holding higher numbers.

WHERE IT DOES BECOME A PROBLEM 

Unless you use some additional witchcraft, you functionally have a 2.14 billion cap on how high your cell number can go, positive or negative. The GPU will start resetting its numbers if you go past 2.14 billion, whereas the CPU will handle it fine, because it can count far higher. This reset wouldn't cause anything to crash, but it would cause your CPU and GPU to diverge in what they produce.

Is this a problem? Well... probably not. One 2.14 billionth of Earth's diameter is about six millimeters, or about a quarter of an inch. Creating a procedurally generated universe that requires more unique-per-cell detail than that is outside the scope of this book.

NOW THAT THAT'S DONE: A FUNCTION TO BREAK THE HASH RESULT IN HALF 

How do we take a binary number that the CPU is storing in 64 bits and extract the first set of 16 bits, and the second set of 16 bits? Although you can think of this as a single function, for now, let's think of it as two functions.

A FUNCTION FOR THE SECOND/RIGHT SET OF SIXTEEN BITS 

This is probably the easiest one; we've routinely been obliterating the left side of the 32 bit binary numbers via an AND mask:

x &= 0xFFFFFFFF

We can accomplish the same feat with a shorter AND mask:

x &= 0xFFFF

Pretend, for example, you had this 32 bit binary number:

0101 1001 1001 1011 1010 0001 1101 0001

An AND mask with 16 bits of 1's has the effect of chopping the bits off the left side:

0101 1001 1001 1011 1010 0001 1101 0001 input
0000 0000 0000 0000 1111 1111 1111 1111 mask 
---- ---- ---- ---- ---- ---- ---- ---- bitwise AND operator
0000 0000 0000 0000 1010 0001 1101 0001 result 

Thus, we're just doing again something we've done already.

A FUNCTION FOR EXTRACTING THE FIRST/LEFT SIXTEEN BITS 

We're still doing things we've done already for the next challenge, which is to extract the first/left half of the 32 bits. Lets consider that same input number:

0101 1001 1001 1011 1010 0001 1101 0001

If we only care about the left sixteen bits, then a bitshift operator 16 bits to the right will move the left bits right, and the right bits will just fall off the cliff:

0101 1001 1001 1011 1010 0001 1101 0001 input
---- ---- ---- ---- ---- ---- ---- ---- bitshift 16 places: >>16 operator
0000 0000 0000 0000 0101 1001 1001 1011 

WHICH BITS ARE THE "FIRST" BITS?

I've been writing left/first and right/second because I don't think "first" is intuitively obvious. For a lot of people, the "first" digits are the ones on the right that change first when a number ticks upward. Also, a very large portion of the world reads right to left, and we're all using Arabic numerals; modern Arabic itself reads right to left. That said, when people are talking about the "first" or "last" digits of a binary number, they usually seem to be talking about the left digits being the first digits, and the right digits being the last digits.

RETURNING A RESULT FROM EACH FUNCTION 

We still need to normalize the result, which in this situation, means taking a very large random number and squishing it to fit between 0 and 1. The hash we started with drew a number between 0 and 1 by dividing the resulting 32 bit binary number by the total number of possibilities that exist for 32 bit binary numbers, which is roughly 4.29 billion, or specifically:

	var random : float = x / 4294967296.0

Because the binary number is getting split into two numbers with 16 binary digits, that math no longer works. We need to divide by the total number of numbers that can be expressed with 16 bits. 

If you want to reason out the math, we know that two bits, 00 through 11, can express four possibilities, or 2^2. 000 through 111 is 0 through 7, which is eight possibilities. Thus, 16 bits can express 2^16 possibilities. So in order to get our random number between 0 and 1, we divide our 16 bit number by that number of possibilities:

	var result : float = x / pow(2,16)

...or...

	var result : float = x / 65536.0

TAKE A STAB AT WRITING OUT THE FUNCTIONS FOR A RESULTING X VALUE AND A RESULTING Y VALUE 

Or just copy me, I can't stop you.

I have a binary integer that's been spit out of the hash. For my normalized x hash result, I'll use the left/first digits.

	func normalized_x_hash_half(input: int) -> float:
		input = input >> 16
		var result : float = input / 65536.0
		print("The x half is " + str(result))
		return result

And here's a function for the y portion, which I'm drawing from the right half of the binary number: 

	func normalized_y_hash_half(input: int) -> float:
		input &= 0xFFFF
		var result : float = input / 65536.0
		print("The y half is " + str(result))
		return result

A WARNING ABOUT THAT DIVISION LINE 

If you happen to only write out 65536 instead of 65536.0, then the math won't work out, and will return either a 0 or 1. That's because when you do division with integers, the fractional part gets discarded. It actually doesn't matter that we've pre-arranged a float on the left to accept the result--the fractional part will get destroyed before it gets there, and obtaining the fractional part is literally the reason we're here. So: Write it out as a float to force it to work as float math. Fortunately, Godot actually will warn you about this with a yellow flag--just not a red one. But it will break your shader, so treat it as a red flag.

	func wellons_integer_hash(input: int) -> int:

And then copy and paste almost everything verbatim into it. I'll tweak the top and the bottom lines, however, so look at the second and the second to last lines of this block:

	func wellons_integer_hash(input: int) -> int:
		var x = input
		print(str(x) + " is the input integer.")
		print(to_binary(x) + " is the initial input in binary.")
		print(to_binary(x>>16) + " is X bitshifted.")
		x ^= x >> 16
		print(to_binary(x) + " the XOR result of the input and bitshifted input")
		x *= 0x7feb352d # 	2,146,121,005 in hex form,
		#					0111 1111 1110 1011 0011 0101 0010 1101 is binary
		print(to_binary(x) + " after getting multiplied by magic constant number one")
		x &= 0xFFFFFFFF # A chain of 32 ones in binary
		print(to_binary(x) + " after masking out the numbers on the left half, if any exist")
		print(to_binary(x>>15) + " after a bit shift 15 spaces right")
		x ^= x >> 15
		print(to_binary(x) + " the XOR result of the previous two values.")
		x *= 0x846ca68b # 	2221713035 in hex form, and in binary...
		#					1000   0100   0110   1100   1010   0110   1000   1011
		print(to_binary(x) + " after multiplying by the second magic constant")
		x &= 0xFFFFFFFF
		print(to_binary(x) + " after masking out the numbers on the left half, if any exist")
		print(to_binary(x>>16) + " the third and final bitshift, sixteen places to the right")
		x ^= x >> 16
		print(to_binary(x) + " XOR on the previous two values")
		var hashed_integer_result = x
		return hashed_integer_result

EFFICIENTLY CHAINING IT ALL TOGETHER

Now, my initial plan was to do the following:

	Take an X and Y input, and
	Shove that result into the "combine_x_and_y_to_uint" function, and
	Shove that result into the wellons_integer_hash function, and
	Shove that result into the normalized_x_hash_half function, and finally...
	...do the entire spiel over again for the y half? What?!

But that presents a problem. That means running the hash twice. You know what everyone hates about procedural terrain generation? Waiting. Time is money. Life is short. Much too short to wait on two hash functions. How do we handle this?

CREATE A FUNCTION THAT RETURNS TWO RESULTS?

Well... yes? I mean, it's not technically allowed to "return" two results, but it's certainly allowed to *do* something with two results. All we care about here is that gdscript prints the finished randomized x and y result to the terminal. We're doing the exact same thing on the GPU. That means whatever function on the GPU is called to do the visual equivalent of print() will have to bundle the work together in one function.

So let's comment out our original x and y functions, and take a stab at something that prints the results of both in one pass:

	func print_x_and_y_halves(input: int) -> void:
		var x : int = input >> 16 #	bitshift 16 places to the right, capturing 
		#						the left 16 bits of the binary number
		var y : int = input & 0xFFFF #eliminate the left 16 bits of the binary number, preserving
		#								capturing the right sixteen bits only.
		var x_float : float = x / 65536.0
		var y_float : float = y / 65536.0
		print("The random x coordinate is " + str(x_float) + " and the random y coordinate is " + str(y_float))
	
NOW WE CAN STRING THEM TOGETHER 

Our pseudocode now only requires one pass: 

	Take an X and Y input, and
	Shove that result into the "combine_x_and_y_to_uint" function, and
	Shove that result into the wellons_integer_hash function, and
	Shove that result into the print_x_and_y_halves function. Party!

Return to the ready() function. I'll be writing it with a lot of extra variables, but if you like being succinct, go for it.

	func _ready() -> void:
		var x : int = 1
		var y : int = 3
		var combined_x_and_y = combine_x_and_y_to_uint(x, y)
		var combo_hashed = wellons_integer_hash(combined_x_and_y)
		print_x_and_y_halves(combo_hashed)

OTHER DIRECTIONS 

You should know enough about bitwise operators to take this into the Z direction as well, if you wanted to generate three dimensional noise. This is sometimes called a "hypertexture". But note that just because noise is two dimensional doesn't mean that it can't be used in three dimensions--that's extremely common, and how most water shaders and terrain generators operate. Truly three dimensional noise is more for things like clouds, where we actually care about particle density in all three dimensions. For procedural terrain, you'd more likely see it where the terrain itself was actually three dimensional--caves, floating islands, and so forth.
