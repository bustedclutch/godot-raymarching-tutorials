I ended the last chapter with a problem: The hash functions accept single numbers. And our hash function in particular only accepts unsigned integers.

Let's restate the issue as pseudocode that we can iterate on:

	Each input X and Y will have to:

		Individually get multiplied by a different odd, large constant, and
		Combined with each other via an XOR operation (the caret! ^), and
		Fed into the hash now that they are one binary number, and
		Split in two at the end, giving you two 16 bit numbers as output X and Y.

We're still in gdscript territory. We are still sticking, desperately, to abstracting out these functions as much as we can. The current hash function is just a bunch of code sitting under the ready() function, which honestly, I'm not bothered by.

But to prepare ourselves for rewriting this in shader code, let's keep the balance abstract.

We need a function, that takes two x and y values as inputs, and produces a single combined integer output that's usable for a hash function. That gives us something close to a header and a return:

	function vec2_to_uint_converter(vec2 input) -> integer

If you're paying attention, I've introduced a terrible bug already. More on that later, in the section about the terrible error.

We know x and y each need to be multiplied by a big number; I'll just write out x for now and plan to write y after I've figured x out:

	variable x = input.x 
	x *= big_hexadecimal_number

We know multiplying x by a big number exposes us to the possibility that x will become a very large number expressed in more than 32 bits, because the CPU will do math all the way up to 64 bits by default. We don't want that, so we'll mask it. We know that "F" means 1111 in hexadecimal, and that an AND operation on a 32 bit number that's all 1's will functionally drop a bigger number down to 32 bits. This keeps our GPU math and CPU math from diverging.

	x &= 0xFFFFFFFF

Finally, we know that we can combine x and y using an additional XOR operation. 

	variable result = x ^ y 

This gives us some not-quite-right-looking code to clean up and make pretty. 

	function vec2_to_uint_converter(vec2 input) -> integer
		variable x = input.x 
		x *= big_hexadecimal_number
		x &= 0xFFFFFFFF
		variable result = x ^ y

...a few beautification steps first:

	func vec2_to_uint(xy : Vector2) -> int: #This line is broken; we'll find out why later
		var x = xy.x
		var y = xy.y
		x *= 0x27d4eb2d #multiply by a big random odd constant
		y *= 0x165667b1 #multiply by some other big random odd constant
		x &= 0xFFFFFFFF
		y &= 0xFFFFFFFF
		return x ^ y #return x combined with y using XOR

IT'S HERE THAT IT DAWNS ON ME THAT I'VE MADE A TERRIBLE MISTAKE 

All of this bitwise math is going to break under the hood.

If you haven't figured out why, ponder why! I never actually told you, so, don't feel bad if you have no idea.

WE SHOULD NEVER HAVE ACCEPTED A VEC2 AS THE INPUT 

Vector2's store their individual components as floats. Floats are not stored in the same way that integers are. In the binary number system, when you have floats, an arbitrary number of bits are reserved for what's called the "mantissa" and another part called the "exponent". Mantissa sounds like a cocktail you'd get at a bar or a campy name for a James Bond villain. So we'll stick with mantissa. We've already heard about the bit that's used to sign the integer, which is the far left bit called the "sign bit". Claude likens the exponent part to a magnifying knob that dials up the size of the mantissa. 

Sign---	Exponent---	Mantisssssssssssahhhhhhhhhhh!
0		0111 1111 	001 0000 0000 0000 0000 0000 

This will screw up our bitwise calculations for obvious reasons because these might as well be two different languages.

So: We should avoid contaminating our function with floats, as they'll introduce a completely different set of binary numbers, causing silent problems.

A refactoring (and renaming!) of the function is thus in order. Correct the first line, and the declaration of x and y:

	func combine_x_and_y_to_uint(x: int, y: int) -> int:
		x *= 0x27d4eb2d #multiply by a big random odd constant
		y *= 0x165667b1 #multiply by some other big random odd constant
		x &= 0xFFFFFFFF
		y &= 0xFFFFFFFF
		return x ^ y #return x combined with y using XOR

BUT WAIT: CAN'T THE SIGNED PORTION OF THE BIT SCREW EVERYTHING UP?

No. Not with the masking. Read below if you care about why.

On the GPU, we'll have access to a built-in function, uint(). It doesn't actually change the first bit from 1 to 0. It just causes the binary number to be interpreted differently. This binary number:

1111 1111 1111 1111 1111 1111 1111 1111  (also called 0xFFFFFFFF in hexadecimal)

...is interpreted as -1 normally if it's considered signed, but if you treat it as an unsigned integer, then it's interpreted as 4,294,967,295. The binary number stays the same.

We don't have a uint() in place here. So could that mess us up? Well, let's just take an example. Say I want to use x= -1. Using uint(x) on the GPU will relabel the binary number, but the binary number will still be that long string of ones.

But: What does -1 look like if you're in 64 bits? Well, I'll break it up into two lines, but it looks like this:

  hex:     0xFFFFFFFFFFFFFFFF

  first half:  1111 1111 1111 1111 1111 1111 1111 1111  
  second half: 1111 1111 1111 1111 1111 1111 1111 1111
  decimal (unsigned interpretation): 18,446,744,073,709,551,615   (= 2⁶⁴ − 1)
  decimal (signed interpretation):   −1

What about -3? In 64 bits?

  hex:     0xFFFFFFFFFFFFFFFD

  first half:  1111 1111 1111 1111 1111 1111 1111 1111 
  second half: 1111 1111 1111 1111 1111 1111 1111 1101

  unsigned interpretation:  18,446,744,073,709,551,613   (= 2⁶⁴ − 3)
  signed interpretation:   −3

So far, we're noticing a pattern: The first half is always a bunch of ones, and we chop those with a mask like we did in the hash function. That pattern holds past the 2.14 billion mark, which is when the GPU's integer length effectively stops it from holding higher numbers.

WHERE IT DOES BECOME A PROBLEM 

Unless you use some additional witchcraft, you functionally have a 2.14 billion cap on how high your cell number can go, positive or negative. The GPU will start resetting its numbers if you go past 2.14 billion, whereas the CPU will handle it fine, because it can count far higher. This reset wouldn't cause anything to crash, but it would cause your CPU and GPU to diverge in what they produce.

Is this a problem? Well... probably not. One 2.14 billionth of Earth's diameter is about six millimeters, or about a quarter of an inch. Creating a procedurally generated universe that requires more unique-per-cell detail than that is outside the scope of this book.

NOW THAT THAT'S DONE: A FUNCTION TO BREAK THE RESULT IN HALF 

Our hash function currently divides x by the total number of possibilities, 4.29 billion-ish, to produce a result between 0 and 1. But before it does that, there's still a very random binary number that's 32 bits long and which we could split in half. That would give us two sixteen bit binary numbers. In hex, that's 0xFFFF, or:

1111 1111 1111 1111 

That's 2^16 - 1. 

If you want to reason that out in your head in the future, "11" in binary is 2^2-1, or 4-1=3.
