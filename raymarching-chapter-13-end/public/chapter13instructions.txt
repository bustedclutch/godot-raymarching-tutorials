	vec3 random_point_light(vec2 xy, float local_scale, uint hashed_uint, float x_grid_id, float y_grid_id){
		uint combined_hash_uint = hashed_uint;
		uint x_uint = combined_hash_uint >> 16u;
		uint y_uint = combined_hash_uint &0xFFFFu;
		float x_normalized_hash = float(x_uint) / float(65536);
		float y_normalized_hash = float(y_uint) / float(65536);
		vec2 coordinate_pos_in_global_space = vec2(x_grid_id+x_normalized_hash,y_grid_id+y_normalized_hash);
		vec2 scaled_current_position = xy; // * local_scale;
		float current_distance_to_point = abs(distance(coordinate_pos_in_global_space, scaled_current_position));
		vec3 light = vec3(step(current_distance_to_point, 0.08));
		return light;
	}

	uint combined_x_and_y_uint(uint x, uint y){
		x *= 0x27d4eb2du ; //multiply by a big random odd constant
		y *= 0x165667b1u ; //multiply by some other big random odd constant
		x &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		y &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		return x ^ y ; //return x combined with y using XOR
	}

	uint wellons_integer_hashed(uint input){
		uint x = input;
		x ^= x >> 16u;
		x *= 0x7feb352du;
		x &= 0xFFFFFFFFu;
		x ^= x >> 15u;
		x *= 0x846ca68bu;
		x &= 0xFFFFFFFFu;
		x ^= x >> 16u;
		uint hashed_integer_result = x;
		return hashed_integer_result;
	}

	vec3 grid_line (vec2 xy, float local_scale) {
		//Take fractional parts of X and Y
		float x_fract = fract(xy.x);
		float y_fract = fract(xy.y);
		//Test each part to see if the fractional part is close to zero or almost one.
		float grid_test = step(x_fract, 0.02) + step(0.98, x_fract) + step(y_fract, 0.02) + step(0.98, y_fract) ;
		float lit = min(1.0, grid_test);
		vec3 grid_light = lit * vec3(0.0, 0.5, 0.2);
		return grid_light;
	}

	vec2 uv_math(vec2 xy, float local_scale){
		vec2 uv = xy;
		uv = uv * 2.0 - 1.0;
		uv = uv * vec2(1.0, -1.0);
		uv = uv * local_scale;
		return uv;
	}

REMEMBERING WHAT HAPPENS

It's been a few days since I wrote the last chapter, and I've since completely forgotten everything that's going on. In terms of tracing what happens in a shader, you'd normally start with the vertex() shader, and then the fragment() shader, but there's no vertex() action yet. So to trace the events of this function, read the fragment() function from top to bottom. I reiterate its happenings below.

First, the (lowercase) uv variable gets set through the uv_math() function. It takes the (uppercase) UV constant and the scale uniform as arguments.

Then, a vec3 that will make up the shader's evolving color, "col", is initialized as black vec3(0.) .

Then, the "uv_grid_id_x" is established by calling a helper function, "grid_id_x". Very few shaders would go to the trouble to write a helper function out to determine the grid ID, since it's just a floor() of the current uv, but I did.

The same happens with "uv_grid_id_y". As a reminder, this is the identifier for the grid cell we're currently in.

After that, the X and Y grid IDs are combined into a single unsigned integer, a uint, using the combined_x_and_y_uint() function, and assigned to the variable "combined_uv_for_hashing".

It's then tossed into the wellons_integer_hashed() function and the result is assigned to the uint "uv_hash".

Here, the random_point_light() function does a lot of work--arguably too much. It divides the hash into two uints, and divides that result by the total number of uints possible along the X axis and the Y axis, to provide a random X and Y value. That value is between 0 and 1 for each, and it gets added to the X and Y grid ID to find a random point in global space for the grid cell we're in presently. The distance to that point, from our current global uv location, is run through a step function to decide whether to pass a 1.0 out of the function to light the pixel.

Finally, the grid lines get added, and the completed color gets assigned back to ALBEDO.

LET'S CRITIQUE THIS METHOD

I see a couple issues in the code so far. The first issue is that it produces significant aliasing problems. I'm choosing not to care about those right now, because the current shading of the points is just a proof-of-concept to confirm that we are, actually, randomly putting points down, and that the random points match our gdscript function.

The second problem runs deeper. The random point is initially an X and Y position between 0 and 1, and then the grid IDs are added back in, taking the random point from "local" space to global space. A random position of (0.35, 0.21) gets added to a grid ID of something like (2,4) to produce a global position for the random point at (2.35,4.21). The distance calculations then compare the distance between that local point and somewhere else, something like (2.9, 4.9).

This remains true even if the cell and its random point are hundreds of millions of units away. And that might feel a little silly, since even in that circumstance, the distance between the random point and the present uv coordinate is still small, like, less than a single unit typically.

You can see it described in the function as "scaled_current_position" and "coordinate_pos_in_global_space". Even if we were in the ten-million by ten-millionth grid cell, we're running the distance calculation in global space, confirming that the current uv coordinate and the random point ten million units from the origin are something like 0.5 units apart.

That math might sound unnecessary, because it is. We can ask the points to evaluate their local distances before converting them to global space, if we even need the global space coordinates at all. This "we don't actually care about global space here" spin on Worley noise is credited to Inigo Quilez. If this sounds confusing and hard to understand, it'll hopefully make more sense as we code it.

It's probably the only update I want to give to the underlying function.

I'M ASSUMING YOU ALREADY KNOW ABOUT SCOPES

Scopes "are an area of code where you can access a variable", including "local scope/function scope, global scope, block scope, and nested scope" per Saldina Nurak a.k.a. CodeBeauty who has a nice video about scope titled "Variable scopes w/ examples (Local vs Global vs Block vs Nested) at:

https://www.youtube.com/watch?v=mvAYOcLr8Dk

A brief summary: A variable declared inside of a function can't be accessed outside of that function. This is a "locally scoped" variable.

We don't really deal too much with scope in shaders because nested functions aren't even allowed--you can't shove a function inside of the fragment() function. Also, the functions I've written so far have always returned something--they are not "void" functions that act without returning something like a float or a vector.

And although we've used "globally scoped" variables, in the shader context, that would technically mean a variable declared outside of any of the shader's functions. You might have noticed those variables fall into two categories: Constants, which the shader can't change, and Uniforms shipped in by the CPU, which the shader also can't change.

For the current chapter, we'll be structuring the functions using the "inout" parameter modifier. It will allow local variables, declared inside of fragment(), to get altered by functions outside of fragment(). 

This gives us a path around a scope problem that could otherwise be very annoying. We'll be using it a bunch shortly, so if the concept of scope itself is new to you, pop open the video, listen to Saldina Nurak explain it to you, and come back.

THE "WINNER'S PODIUM" FUNCTION 

I'm not winning any awards for mastery of if-else logic, but this function seems more important than my previous follies.

The declaration is obvious: It should accept a distance as the first argument, and the worley_distances[] array will always go in as the second.

Crucially, it'll also need the "inout" keyword--otherwise, the function will not have the ability to modify the distance array. Without the keyword, it'll silently bug out, and never modify the array.

	void winners_podium(float input_distance, inout float distance_array[5]){
		
	}

If you're wondering whether you could have written the "[5]" after "float" instead of after "distance_array", well, it seems that Godot doesn't care, and in terms of what's better, I'm the wrong person to ask. Onward!

In terms of the logic, it could compare the input_distance to the current winner first, or it coudl compare it to the current fifth place finisher. Which should it be? Well, if we start by comparing it to the fifth place finisher, we risk having the input replace the fifth place finisher even when it should place higher--or alternatively, we'd have tons of nested logic to test upwards.

So the logical place to begin the if-else analysis is at the very top, and to work downward:

	void winners_podium(float input_distance, inout float distance_array[5]){
		if (input_distance < distance_array[0]){
			distance_array[0]=input_distance;}
		else if (input_distance < distance_array[1]){
			distance_array[1]=input_distance;}
		else if (input_distance < distance_array[2]){
			distance_array[2]=input_distance;}
		else if (input_distance < distance_array[3]){
			distance_array[3]=input_distance;}
		else if (input_distance < distance_array[4]){
			distance_array[4]=input_distance;}
		}

This code has a significant bug: It leaves all the finishers lower on the podium in place, when they should all be demoted to make room. So, write in the demotions:

WHERE ARE WE IN LOCAL SPACE FUNCTION 

Althoug this function might not be necessary, it's got similar vibes to a pair of functions we already wrote:

	uint grid_id_x(vec2 xy){
		float x_vec = xy.x;
		float x_floor = floor(x_vec);
		return uint(int(x_floor));
	}
	uint grid_id_y(vec2 xy){
		float y_vec = xy.y;
		float y_floor = floor(y_vec);
		return uint(int(y_floor));
	}

This returned the unsigned integer floor() of X and Y, which is the opposite of what we want. If we're in the ten millionth grid cell of X, such that we're at 10,000,000.50 on the X axis, we just want the 0.50 part, to bring us into the neighborhood we actually care about. 

The code for this is simple:


	vec2 local_space_uv_coordinate(vec2 xy){
		return vec2(fract(xy.x), fract(xy.y));
	}


FLEXIBLE RETURN POINT OR POINTS FOR THIS CELL FUNCTION 

Claude pointed out that there are two interesting options here:

	Pattern A: fixed-size output + count.
	void points_for_cell(uvec2 cell_id, out vec2 points[5], out int count);
	The function always allocates space for the max (5), fills however many it actually generates, and
	reports the count via out int. The caller does for (int i = 0; i < count; i++). This is the textbook
	way to handle "0 to N" in languages without dynamic arrays. It works fine, it's predictable, and it
	scales to Poisson cleanly.

	Pattern B: pass the podium in and let the function update it directly.
	void add_cell_contributions_to_podium(uvec2 cell_id, vec2 current_pos, inout float
	distance_array[5]);
	Instead of returning points at all, the function computes whatever number of random points it wants
	for that cell, measures distances against current_pos, and calls winners_podium internally. The "how
	many points" question never leaves the function. This is probably the most flexible pattern for
	Worley specifically, because the caller doesn't need to know or care.

	The tradeoff between them: Pattern A separates "where are the points" from "what do I do with them" —
	better if you also want, say, a debug visualization that just draws the points. Pattern B is tighter
	and avoids ever materializing a points array, but couples the point-generator to the podium API.

	For a tutorial, I'd lean Pattern A. The count-based idiom is genuinely useful pedagogy — it shows up
	everywhere in graphics code (collision results, query hits, etc.) and is worth teaching once
	explicitly. Pattern B is the "experienced GLSL author" version you could mention as a "by the way,
	you could also do this" aside.

	The thing to abandon is the idea of returning a variable-size array. GLSL just won't let you, and any
	workaround (sentinel values, etc.) is worse than picking one of the above.

I've committed to limiting the length of my chapters, and Claude did a good explanation of the more flexible options so: I'm going to code an *inflexible, one point* version. Besides, I'm trying to follow my own advice, which is to drive the fastest, simplest, easiest path to something visible on screen, and then layer the complexity on afterward.

We'll be doing a little cut and paste from last chapter's fragment() and random_point_light() functions to build a function that only generates one random point when given a grid cell id:

	vec2 random_point(uint grid_id_x, uint grid_id_y){
		uint combined_uint_for_hashing = combined_x_and_y_uint(grid_id_x, grid_id_y);
		uint combined_hash_uint = wellons_integer_hashed(combined_uint_for_hashing);
		uint x_uint = combined_hash_uint >> 16u;
		uint y_uint = combined_hash_uint &0xFFFFu;
		float x_normalized_hash = float(x_uint) / float(65536);
		float y_normalized_hash = float(y_uint) / float(65536);
		return vec2(x_normalized_hash, y_normalized_hash);
	}

It's worth noting what the above function does *not* do. It only returns a fractional, 0-1 vec2 for the current cell you're in. We'll eventually be measuring the distance from a place in the current cell to a place in the neighboring cell. 

I use a lot of office building metaphors--if a grid cell is an office building, every grid cell has a point in the middle at (0.5,0.5). If there's a cubicle number 5 on the fifth floor of an office building, there's an identical cubicle in every other office building. So we'll need to make sure our distance() function takes into account the position of the neighbors. We don't want to run distance((0.5),(0.5,0.5)) as that would erroneously return zero, when we likely intended the cubicle in the center, compared to the cubicle one building over. We'll handle that in the loop function.
