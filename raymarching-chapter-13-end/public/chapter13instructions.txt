	uint combined_x_and_y_uint(uint x, uint y){
		x *= 0x27d4eb2du ; //multiply by a big random odd constant
		y *= 0x165667b1u ; //multiply by some other big random odd constant
		x &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		y &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		return x ^ y ; //return x combined with y using XOR
	}

	uint wellons_integer_hashed(uint input){
		uint x = input;
		x ^= x >> 16u;
		x *= 0x7feb352du;
		x &= 0xFFFFFFFFu;
		x ^= x >> 15u;
		x *= 0x846ca68bu;
		x &= 0xFFFFFFFFu;
		x ^= x >> 16u;
		uint hashed_integer_result = x;
		return hashed_integer_result;
	}

REMEMBERING WHAT HAPPENS

It's been a few days since I wrote the last chapter, and I've since completely forgotten everything that's going on. In terms of tracing what happens in a shader, you'd normally start with the vertex() shader, and then the fragment() shader, but there's no vertex() action here. So to trace the events of this function, read the fragment() function from top to bottom. I reiterate its happenings below.

First, the (lowercase) "uv" variable gets assigned through the uv_math() function. It takes the (uppercase) UV constant and the scale uniform as arguments.

Then, a vec3 that will make up the shader's evolving color, "col", is initialized as black vec3(0.) .

Then, the "uv_grid_id_x" is established by calling a helper function, "grid_id_x". Very few shaders would go to the trouble to write a helper function out to determine the grid ID, since it's just a floor() of the current uv, but I did.

The same happens with "uv_grid_id_y". As a reminder, this is the identifier for the grid cell we're currently in.

After that, the X and Y grid IDs are combined into a single unsigned integer, a uint, using the combined_x_and_y_uint() function, and assigned to the variable "combined_uv_for_hashing".

It's then tossed into the wellons_integer_hashed() function and the result is assigned to the uint "uv_hash".

Here, the random_point_light() function does a lot of work--arguably too much. It divides the hash into two uints, and divides that result by the total number of uints possible along the X axis and the Y axis, to provide a random X and Y value. That value is between 0 and 1 for each, and it gets added to the X and Y grid ID to find a random point in global space for the grid cell we're in presently. The distance to that point, from our current global uv location, is run through a step function to decide whether to pass a 1.0 out of the function to light the pixel.

Finally, the grid lines get added to the shader, if there's any nearby, and the completed color gets assigned back to ALBEDO.

LET'S CRITIQUE THIS METHOD

I see a couple issues in the code so far. The first issue is that it produces significant aliasing problems. I'm choosing not to care about those right now, because the current shading of the points is just a proof-of-concept to confirm that we are, actually, randomly putting points down, and that the random points match our gdscript function.

The second problem runs deeper. The random point is initially an X and Y position between 0 and 1, and then the grid IDs are added back in, taking the random point from "local" space to global space. A random position of (0.35, 0.21) gets added to a grid ID of something like (2,4) to produce a global position for the random point at (2.35,4.21). The distance calculations then compare the distance between that global point and the global version of our current position, like (2.9, 4.9).

This remains true even if the cell and its random point are hundreds of millions of units away. And that might feel a little silly, since even in that circumstance, the distance between the random point, and the present uv coordinate, is still small, like, less than a single unit typically.

You can see it described in the function as "scaled_current_position" and "coordinate_pos_in_global_space". Even if we were in the ten-million by ten-millionth grid cell, we're running the distance calculation in global space, confirming that the current uv coordinate, and the random point, both possibly ten million units from the origin, are something like 0.5 units apart.

That math might sound unnecessary, because it is. We can ask the points to evaluate their local distances from each other before converting them to global space, if we even need the global space coordinates at all. This "we don't actually care about global space here" spin on Worley noise is credited to Inigo Quilez. If this sounds confusing and hard to understand, it'll hopefully make more sense as we code it.

It's probably the only substantive overhaul I want to give to the underlying function's logic.

I'M ASSUMING YOU ALREADY KNOW ABOUT SCOPES

Scopes "are an area of code where you can access a variable", including "local scope/function scope, global scope, block scope, and nested scope" per Saldina Nurak a.k.a. CodeBeauty who has a nice video about scope titled "Variable scopes w/ examples (Local vs Global vs Block vs Nested) at:

https://www.youtube.com/watch?v=mvAYOcLr8Dk

A brief point: A variable declared inside of a function can't be accessed outside of that function. This is a "locally scoped" variable.

We don't really deal too much with scope in shaders because nested functions aren't even allowed--you can't shove a function inside of the fragment() function. Also, the functions I've written so far have always returned something--they are not "void" functions that act without returning something like a float or a vector.

And although we've used "globally scoped" variables, in the shader context, that would technically mean a variable declared outside of any of the shader's functions. You might have noticed those variables fall into two camps: Constants, which the shader can't change, and Uniforms shipped in by the CPU, which the shader... also can't change.

So if we need a variable that can bounce among the functions and be changed, what do we do?

For the current chapter, we'll be structuring functions using the "inout" parameter modifier. It will allow local variables, declared inside of fragment(), to get altered by functions outside of fragment(). 

This gives us a path around a scope problem that could otherwise be very annoying. We'll be using it a bunch shortly, so if the concept of scope itself is new to you, pop open the video, listen to Saldina Nurak explain it to you, and come back.

FLEXIBLE RETURN POINT OR POINTS FOR THIS CELL FUNCTION 

For the function that provides the random point for any given cell, I initially planned to write a flexible one that could return multiple points. I was a bit bewildered on how I wanted to frame it, so, I posed the question to Claude the chatbot.

Claude pointed out that there are two interesting options here:

	Pattern A: fixed-size output + count.
	void points_for_cell(uvec2 cell_id, out vec2 points[5], out int count);
	The function always allocates space for the max (5), fills however many it actually generates, and
	reports the count via out int. The caller does for (int i = 0; i < count; i++). This is the textbook
	way to handle "0 to N" in languages without dynamic arrays. It works fine, it's predictable, and it
	scales to Poisson cleanly.

	Pattern B: pass the podium in and let the function update it directly.
	void add_cell_contributions_to_podium(uvec2 cell_id, vec2 current_pos, inout float
	distance_array[5]);
	Instead of returning points at all, the function computes whatever number of random points it wants
	for that cell, measures distances against current_pos, and calls winners_podium internally. The "how
	many points" question never leaves the function. This is probably the most flexible pattern for
	Worley specifically, because the caller doesn't need to know or care.

	The tradeoff between them: Pattern A separates "where are the points" from "what do I do with them" —
	better if you also want, say, a debug visualization that just draws the points. Pattern B is tighter
	and avoids ever materializing a points array, but couples the point-generator to the podium API.

	For a tutorial, I'd lean Pattern A. The count-based idiom is genuinely useful pedagogy — it shows up
	everywhere in graphics code (collision results, query hits, etc.) and is worth teaching once
	explicitly. Pattern B is the "experienced GLSL author" version you could mention as a "by the way,
	you could also do this" aside.

	The thing to abandon is the idea of returning a variable-size array. GLSL just won't let you, and any
	workaround (sentinel values, etc.) is worse than picking one of the above.

We'll eventually be measuring the distance from a place in the current cell to a place in the neighboring cell. I use a lot of office building metaphors--if a grid cell is an office building, every grid cell has a point in the middle at (0.5,0.5). So we'll need to make sure our distance() function takes into account the position of the neighbors. We don't want to run distance((0.5,0.5),(0.5,0.5)) as that would erroneously return zero, when we likely intended the cubicle in the center, and a cubicle one building over. ABOUT THAT SHADOWING PROBLEM 

I'm doing an unfortuante beginners thing, which is to shadow the name of functions and parameters. "grid_ix_x" shows up as both the function name and the parameter name where that function's result gets used.

To avoid this, you can rewrite your functions and parameter names so that functions get verbs, and values get nouns. Thus, grid_id_x() could be refactored to get_grid_id_x(). (Other options include: Using a verbose word for the function, and a shorter word for the value, or writing "in_" or "arg_" as a prefix to indicate where its an input/argument.) FOLLOW STEVE WORLEY'S ADVICE 

So Steve Worley says that you're much better off just firing a zillion random numbers in to see the results. This would give you a function like this, where each random number, five different Q's below, are factors of the F values, and they're all added together, like this:

	result = Q1*F1 + Q2*F2 + Q3*F3 + Q4*F4 + Q5*F5

A problem arises: This number could be negative, and will be, a lot of the time. Easily fixed, we just take the absolute value of all of them:

	result = abs(Q1*F1 + Q2*F2 + Q3*F3 + Q4*F4 + Q5*F5)

We have another problem, which is that these values are probably going to go way over 1, which would leave us with a white image. We need to normalize somehow. We can't normalize in a conventional way in a shader, but we can divide the result by a number that increases as the UV of x increases from 0 to 1:

	result / (UV.x*normalization_factor)

That normalization_factor can be a uniform sent by gdscript, so that you can tweak that independently.

We might also want a power function. That can magnify the impact of a shader, allowing you to make otherwise faint effects pop:

	pow(x ,n) // x raised to n 
	pow(result, exponent_factor) //with the proposed uniform name
