TWO ROADS DIVERGED 

As introduced by Steve Worley, there wasn't just one point per grid-square. There was a range; sometimes many points, sometimes none. This was a "Poisson" distribution. It was also in 3D, and the code had various additions to increase its efficiency. It was more complex than our shader in almost every respect, except for the fact that it used a simple hash rather than an integer one.

If the chapters are feeling too slow for you, you could make it more hardcore by introducing the Poisson distribution immediately, or by adding tests to make the code more efficient, or by coding it in three dimensions rather than two. For now, I'm going to take the path of inefficient, Poisson-free, two dimensional simplicity.

CLEANING THINGS UP

The worley_points.gdshader file is feeling a little clouded with junk. I duplicated it as "worley_p_start.gdshader" and took a cleaver to its unused sections.

	shader_type spatial;
	render_mode unshaded;

	uniform float scale = 3.0;

	uint grid_id_x(vec2 xy){
		float x_vec = xy.x;
		float x_floor = floor(x_vec);
		return uint(int(x_floor));
	}
	uint grid_id_y(vec2 xy){
		float y_vec = xy.y;
		float y_floor = floor(y_vec);
		return uint(int(y_floor));
	}
		
	vec3 random_point_light(vec2 xy, float local_scale, uint hashed_uint, float x_grid_id, float y_grid_id){
		uint combined_hash_uint = hashed_uint;
		uint x_uint = combined_hash_uint >> 16u;
		uint y_uint = combined_hash_uint &0xFFFFu;
		float x_normalized_hash = float(x_uint) / float(65536);
		float y_normalized_hash = float(y_uint) / float(65536);
		vec2 coordinate_pos_in_global_space = vec2(x_grid_id+x_normalized_hash,y_grid_id+y_normalized_hash);
		vec2 scaled_current_position = xy; // * local_scale;
		float current_distance_to_point = abs(distance(coordinate_pos_in_global_space, scaled_current_position));
		vec3 light = vec3(step(current_distance_to_point, 0.08));
		return light;
	}

	uint combined_x_and_y_uint(uint x, uint y){
		x *= 0x27d4eb2du ; //multiply by a big random odd constant
		y *= 0x165667b1u ; //multiply by some other big random odd constant
		x &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		y &= 0xFFFFFFFFu ; //AND mask to prevent 64 bit CPU from diverging from 32 bit GPU
		return x ^ y ; //return x combined with y using XOR
	}

	uint wellons_integer_hashed(uint input){
		uint x = input;
		x ^= x >> 16u;
		x *= 0x7feb352du;
		x &= 0xFFFFFFFFu;
		x ^= x >> 15u;
		x *= 0x846ca68bu;
		x &= 0xFFFFFFFFu;
		x ^= x >> 16u;
		uint hashed_integer_result = x;
		return hashed_integer_result;
	}

	vec3 grid_line (vec2 xy, float local_scale) {
		//Take fractional parts of X and Y
		float x_fract = fract(xy.x);
		float y_fract = fract(xy.y);
		//Test each part to see if the fractional part is close to zero or almost one.
		float grid_test = step(x_fract, 0.02) + step(0.98, x_fract) + step(y_fract, 0.02) + step(0.98, y_fract) ;
		float lit = min(1.0, grid_test);
		vec3 grid_light = lit * vec3(0.0, 0.5, 0.2);
		return grid_light;
	}

	vec2 uv_math(vec2 xy, float local_scale){
		vec2 uv = xy;
		uv = uv * 2.0 - 1.0;
		uv = uv * vec2(1.0, -1.0);
		uv = uv * local_scale;
		return uv;
	}

	void fragment() {
		vec2 uv = uv_math(UV, scale);
		vec3 col = vec3(0.);
		uint uv_grid_id_x = grid_id_x(uv);
		uint uv_grid_id_y = grid_id_y(uv);
		uint combined_uv_for_hashing = combined_x_and_y_uint(uv_grid_id_x,uv_grid_id_y);
		uint uv_hash = wellons_integer_hashed(combined_uv_for_hashing);
		col = col + random_point_light(uv,scale,uv_hash,floor(uv.x),floor(uv.y));
		col = col + grid_line(uv, scale);
		ALBEDO = col; ALPHA = 1.0;
	}

REMEMBERING WHAT HAPPENS

It's been a few days since I wrote the last chapter, and I've since completely forgotten everything that's going on. The magic happens in the fragment shader, so in terms of reading where the magic happens, you start there.

First, the uv variable gets set through the uv_math() function. It takes the UV constant and the scale as arguments.

Then, a vec3 that will make up the shader's evolving color, "col", is initialized as black vec3(0.) .

Then, the "uv_grid_id_x" is established by calling a helper function, "grid_id_x". Very few shaders would go to the trouble to write a helper function out, but I did.

The same happens with "uv_grid_id_y". As a reminder, this is the identifier for the grid cell we're currently in.

After that, the X and Y grid IDs are combined into a single uint using the combined_x_and_y_uint() function, and assigned to the variable "combined_uv_for_hashing".

It's then tossed into the wellons_integer_hashed() function and the result is assigned to the uint "uv_hash".

Here, the random_point_light() function does a lot of work--it divides the hash into two uints, and divides by the total number of uints possible along the X axis and the Y axis, to provide that random X and Y value. That value is between 0 and 1 for each, and it gets added to the X and Y grid ID to find that actual point in space. The distance to the point is run through a step function to decide whether to pass a 1.0 out of the function to light the pixel.

Finally, the grid lines get added, and the completed color gets assigned back to ALBEDO.

LET'S CRITIQUE THIS METHOD

There's really only one downside to the code so far, which is that it runs the distance function on the literal distance between the points. So, if there's a point in our cell at (2.35,4.21), then it runs the distance between our current point and that point. You can see it described in the function as "scaled_current_position" and "coordinate_pos_in_global_space". Thus, we'd be running this math even if the points were really, really far out. Even if we were in the ten-million by ten-millionth grid cell, we'd be running the calculation in global space, checking the distance between points millions of units away. They'd still be checking each other's distances, even when those ten-million-away units are... literally less than one unit from each other.

That might sound like silly additional math, because it is. We can ask the points to evaluate their local distances before converting them to global space, if we even need the global space coordinates at all. This "we don't actually care about global space here" spin on Worley noise is credited to Inigo Quilez.

It's probably the only update I want to give to the underlying function.

TELL THE RUBBER DUCKY YOUR PROBLEMS 

I have a gut feeling that the code is probably fine for the first couple lines--we do need to figure out what grid cell we're in right now, because everything else emanates from that. So these lines are still good:

	uint uv_grid_id_x = grid_id_x(uv);
	uint uv_grid_id_y = grid_id_y(uv);

But we need to diverge here. As I alluded to before, we don't just want the distance to the closest point inside of our cell, we want the distances to the five closest points. So next, we need to:

	declare five variables to store the five closest distances; we'll follow Worley's nomenclature and name them F1, F2, F3, F4, and F5.

We know shader code is a very, very, very basic machine. We'll be feeding nine different distances in, and we'll need to end up with the distances in the right order. So, we need:

	a function that accepts a new distance, compares it to the existing list of distances, and plops the new one into the right position of the hierarchy, or discards it if it doesn't rank at all 

With that "winner's podium" built, we need winners. So we need to run the race. The race is, of course, determined by the distance from wherever we are right now, so we can add a helper function that answers the "where am I right now?" question. So we'll toss in:

	a helper function that says where we are in local space, even though experienced programmers might not bother to write that out as a separate function

 The distance race must be run not just for the cell we're in, but the cells around our grid. Fortunately, the grid number of the neighbors will be pretty easy to determine--it'll just be the current grid cell, plus or minus one. Cake! So after that, we'll code:

	a function that loops through all nine cells, calculates the location of the point, and then calls the "where are we now" and the "olympics podium" functions to add their respective distances to the rankings

We'll need something pretty to look at. So we'll add:

	a display function, that colors the image based on various combinations of the F1 through F5 values.
