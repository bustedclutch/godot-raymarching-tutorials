LET'S MAKE A GRAPHING CALCULATOR 

Let's return to our original foundation:

	void fragment() {
		vec2 uv = UV;

		uv = uv * 2.0 - 1.0;             // recenter: [0,1] -> [-1,1] on both axes
		uv = uv * vec2(1.0, -1.0);       // flip Y so positive Y points up

		vec3 col = vec3(length(uv));
		col = clamp(col, 0.0, 1.0);
		col = pow(col, vec3(2.2));

		ALBEDO = col; ALPHA = 1.0;
	}

As a reminder, this foundation:
1. Assigns the constant UV to the variable "uv".
2. Recenters the origin from the top-left of the quad to its middle by multiplying by 2 and subtracting 1, so each axis now runs from -1 to 1 instead of 0 to 1.
3. Flips the Y axis with uv = uv * vec2(1.0, -1.0);, so positive Y points up. 
4. Tosses in a distance function as something to show that it's operational, and assigns it to the "col" variable.
5. Fiddles with the colors to account for the human logarithmic perception of color, with the col=pow(col,vec3(2.2)); line. (Warning: This might be done automatically depending on render settings, so keep an eye on the actual look.)
6. Finally, assigns that value to the Godot built-in ALBEDO, which you might recognize from another program as some kind of "out" variable going by another name.

WARNING: The UV centering here is the same as the UV centering in the raymarcher. In the example from many moons ago, I used different math to move the origin from the top left to the bottom left, so that it would look like Shadertoy.

It's worth pointing out that I kept the origin centered to keep things like middle school math, and that's been useful so far, but there will be plenty of times where that gets in the way. I can't say whether you'll always go to the trouble to flip Y, but I don't really do anything in the 2D game space (where that remains the convention), so... I'll always be flipping Y to be up.

GRAPHING A BASIC LINE 

Consider this basic function:
	f(x) = 2x
or, in equation form:
	y = 2x

Although we haven't introduced scale yet, imagine, for a second, you've got a point on your graphing calculator at (3,6). And another point at (1,2), and another point at (4,4). And say we are graphing the straightforward function, y=2x. We need the first two points to light up, and the third point not to light up. Whatever shader function that succeeds in doing that for those three points will probably work for all the others. (This, personally, is how I think through a lot of these problems, though it carries its own hazards, particularly if you don't consider numbers that are negative or otherwise weird.) How do?

One way, and this is where my gut goes, is an if function:

if (2*uv.x == y){
	light_this_pixel = true;
}

But that doesn't really solve our problem, because lines are really small, and pixels are really big. What we really need to know is if we're close to uv.y. How close? Well, "how close" is another way of saying "subtraction", because if I'm ten miles from uv.y, and you're nine miles from uv.y, how close we are is subtracting nine from ten. If you're an axe-murderer and one mile is a comfortable distance, then maybe I don't light the pixel. But if you're the villain from the Texas Chainsaw Massacre, who, per his source material, is able to travel for miles at a very high speed while carrying a chainsaw, then maybe I do want to light the pixel. In any event, here's the revised "proximity" function, which we'll just call "line" for short:

	float line (float x, float y) {
		float f_of_x = 2.0 * x;
		return y - f_of_x;
	}

One more problem to point out--I don't usually care whether the Texas Chainsaw Massacre-er is positive or negative miles away (though I suppose this matters in choosing which direction to run in). In terms of whether the pixel lights up, we just care about the distance. So we can take the absolute value in these functions, in whichever form we choose to write them in:

	float line (float x, float y) {
		float f_of_x = 2.0 * x;
		float raw_distance = y - f_of_x;
		return abs(raw_distance);
	}

Before I move on, I'll point out that you could just as easily shove the abs() method later on in the process. I just want to point out that it will sit somewhere along the chain, and we don't care too much where.

I'll also point out that it's not necessary to abstract this out as a function at all--you could instead do something like this line, though it isn't the path I ultimately took:

	float line = abs(uv.y - 2.*uv.x);

DECIDING WHETHER TO LIGHT THE PIXEL 

Ponder, for a moment, how close is "close" in our current UV space. Our entire graph runs from -1 to 1, so "close enough to light the pixel" is going to be rather small. I just so happen to know that 0.02 will qualify as "close", but you can play with this. 

In the classic "ask the rubber ducky on your desk a question" prose, your shader code needs to ask:
-"am I within 0.02 units of the pixel?" And, 
-"If so, return either 'true' or 1.0," along with 
-"and if I do return 'true' or 1.0, then color the pixel something easy to see."

In case you've forgotten, the step() built-in function is what does this for us.

Claude the chatbot describes the step() function thus:

	step(edge, x) == 0.0   if x < edge
	step(edge, x) == 1.0   if x >= edge

Claude is correct, but in terms of how step actually gets used, you'll notice that not everyone thinks of the function this way.

Rather than thinking of it as "edge" and "the number you're testing", you can also think of the step function as testing whether the second number is greater than or equal to the first number. You could also describe step() this way:

	step(a, b) == 0.0 if b is less than a 
	step(a, b) == 1.0 if b is greater than or equal to a 

Now, I hate this function. Mainly because I constantly mix up the comparison. So, I have mnemonic for it, and since the step function references stairs, and we naturally think of stairs as taking us up, and you're reading this document left to right, then you can think of it as, "if these two variables are stairs, are my stairs going up? If they're taking you up, then 1.0, if they're failing to take you up, then 0.0."

NOW THAT WE ARE THOROUGHLY EMPOWERED WITH MASTERY OF THE STEP() FUNCTION 

...we can actually use it to decide whether to light the pixel. We need to know if our line function is delivering less than 0.02.

	float illuminated_pixel_test = step(line(uv.x, uv.y), 0.02);

In case you're asking, "why aren't we using a boolean?" I suppose the short answer is that booleans are binary and we are artists and artists are virtually always on some kind of spectrum, sexual or otherwise. Some would argue it's just because floats are more useful when doing color math for shaders.

Once we have that illuminated_pixel_test float showing up as 1.0 when our pixel is within 0.02 units from our line, we can color it. We can add, subtract, multiply, divide, and mix vectors (mix is a linear blend). Start with a black background:

vec3 col = vec3(0.0,0.0,0.0);
ALBEDO = col; ALPHA = 1.0;

As a sidebar, since we're preparing to dive deeper into noise functions, we can prepare ourselves to read them by being a bit more brief with our language:

	vec3 col = vec3(0.);
	ALBEDO = col; ALPHA = 1.0;

If you haven't prepared it yet, create a new 3D scene with a plane in Godot. I'm doing it within a new scene with a root Node3D called GraphingCalculator, and a child plane called "BasicGraphingCalculator". That plane's shader is called basic_graphing_calculator.gdshader and the conventions for the rest of this chapter will flow similarly. These are all sandbox objects like the previous sandboxes with noise, so don't obsess about making your names match mine. Just get a plane into the scene and do the surface material override like usual.

Here's what the shader foundation looks like before the math for the line is added:

	shader_type spatial;
	render_mode unshaded;

	void fragment() {
		vec2 uv = UV;
		uv = uv * 2.0 - 1.0;
		uv = uv * vec2(1.0, -1.0);
		vec3 col = vec3(0.);
		ALBEDO = col; ALPHA = 1.0;
	}

And here it is with the math for the line:

	shader_type spatial;
	render_mode unshaded;

	float line (float x, float y) {
		float f_of_x = 2.0 * x;
		float raw_distance = y - f_of_x;
		return abs(raw_distance);
	}

	void fragment() {
		vec2 uv = UV;
		uv = uv * 2.0 - 1.0;
		uv = uv * vec2(1.0, -1.0);

		vec3 col = vec3(0.);

		float illuminated_pixel_test = step(line(uv.x, uv.y), 0.02);
		col = col + illuminated_pixel_test;

		ALBEDO = col; ALPHA = 1.0;
	}

A WARNING I SHOULD HAVE TOLD YOU MUCH SOONER

Note that if I had used that cute one-liner above, then I could have placed the one-liner inside of the fragment function. But nested functions are not allowed, so I have to keep that helper function up-top.

The error that Godot gives, when you do that, is just "expected a comma" or something else entirely unhelpful, it absolutely will not tell you that you put a function in the wrong place.

OTHER OPTIONS

Try replacing your f_of_x with sin(x) or warp its domain by changing it to sin(x+TIME) which will animate it with a sin curve.

Leave it as sin(x) for the next section, because it'll make the next section a lot easier to see and understand.

WHAT ABOUT A GRID? AND SCALE?

Right now, our calculator sees everything in -1 to 1 space, which is fine if we're wanting to stay in normalized device coordinates, but will frustrate us when we want to use our graphing calculator to visualize parts of our noise function. 

I said previously that one method I use to solve a problem is to take a few points on my surface and ask what I would do to those points to create the result I want, and that often leads me to a formula.

So, if we had a point at (0.5,0.5), and a point at (1,1), and we wanted to zoom out to see twice as much so that our visible range went from -2 to +2, what would happen to those two points? And how should we think about zoom anyway, should higher numbers be zooming in, or zooming out? Is zoom even the right word?

Most shaders use "scale" or "frequency", and it's opposite of a day to day zoom lens, so we'll stick with convention and go with scale. So, a -1 to 1 grid that's expanded up to -3,3 would have a scale of 3.

WARNING: I use a centered origin and a -1 to 1 range in my explanations, partly for educational purposes and partly because this is the same coordinate space you'll encounter with normalized device coordinates. But a unit square would either run 0 to 1 or, alternatively, -0.5 to 0.5. This is important, as you'll be pulling noise functions from various people, who typically write their noise functions with a unit square. As you've noticed by now, everything is arbitrary, which means nothing is technically wrong until it fails to match up. "Normalized" seems to have different meanings in different places where I encounter the term, but note that it usually means taking some range of numbers and collapsing it into the range that everything else in your shader uses. Maybe it's a unit square. Maybe it's normalized device coordinates. Maybe it's something else. The normalize() built in function, for example, will take a vector of any length and spit out a vector with a length of 1 that points in the same direction. But the biggest warning I want you to take away is that this will both disturb the scale and cause potentially nonsensical evaluations for your noise function if you use the pattern where the origin is centered. It might work fine. It also might not. I'll try to monitor that situation as we go.

SCALE FIRST 

If I want to show three times as much area in my UV space, instinct tells me it's to either multiply or divide the UV by the scale. (It's multiply.) 

Think again about the y = 2x function. Right now, the UV point at 0,0 measures it at x=0, and the point at 1,0 measures it at x=1. We need to shove more space into this plane, so that the far right edge instead shows us x=3. That allows us to see more, by a factor of 3. So, you just multiply UV by however much the scale is.

The code is thus unexciting:

float scale = 3.0;
uv = uv * scale;

You will only see the difference with the sin curve (more squiggles!) so make sure you've switched your function to show that.

NOW FOR A GRID 

Return to the Texas Chainsaw Massacre step function. If either uv.x or uv.y is close to whatever multiple we want our grid lines to sit at, then it needs to be illuminated, just like the line was illuminated in the illuminated_pixel_test. Take a stab at your options for writing that up. What if it's grid lines on the integers? What if it's grid lines every 0.25 units, or every 5 units? What if you want to abstract that out so you can type any given number of units? 

As a hint, here's what it would look like if I wrote the code inline and not as a separate function, and turned the X grid lines into something:

	//Find the x fractional portion
		float x_fract = fract(uv.x);
	//Test to see if a part is less than 0.02 or greater than 0.98
		float x_tests = step(x_fract, 0.02) + step(0.98, x_fract);
	//If so, light it up! Make sure it can't go over 1 using a min.
		float x_lit = min(1.0, x_tests);
	//Color the light.
		vec3 x_colored_light = x_lit * vec3(1.0, 0.5, 0.0);
		col = col + x_colored_light;

Take a stab at writing it as a function that addresses both X and Y if you haven't already, such that it returns 1.0 as a vec3 when a grid line should appear.

Finally, here's the function in case you want to look at how I did it:

	vec3 grid_line (float uv_x, float uv_y) {
		//Take fractional parts of X and Y
		float x_fract = fract(uv_x);
		float y_fract = fract(uv_y);
		//Test each part to see if the fractional part is close to zero or almost one.
		float grid_test = step(x_fract, 0.02) + step(0.98, x_fract) + step(y_fract, 0.02) + step(0.98, y_fract) ;
		float lit = min(1.0, grid_test);
		vec3 grid_light = lit * vec3(0.0, 0.5, 0.2);
		return grid_light;
	}

Actually calling this function means adding this line to tweak the col variable:

	col = col + grid_line(uv.x, uv.y);

ABSTRACT EVERYTHING

Includes are a glorified copy and paste system, which means if we want to make use of them, we want to abstract as much we can out of the fragment() function. Feel free to refactor this yourself, or follow along as I do it.

One piece will have the line, one piece will have the grid, one piece will have the UV math. The final shader will have the fragment() function and call for all of these pieces.

ABSTRACT THE LINE

Ultimately, you want to just be able to have the col variable add the result of each outsourced function when given the uv coordinate as an input. Our current graphing calculator is most of the way there, though the math for the line light needs to be abstracted out slightly. As of now, it's this:

	float illuminated_pixel_test = step(line(uv.x, uv.y), 0.02);
	col = col + illuminated_pixel_test;

Whereas the function it calls is this:

	float line (float x, float y) {
		//float f_of_x = 2.0 * x;
		float f_of_x = sin(x);
		float raw_distance = y - f_of_x;
		return abs(raw_distance);
	}

We don't want the step() math living in the fragment() function anymore. And instead of a float, it would be more useful to get a vector--that way if the line team decides they want to send a colored line to us, they can.

Here's how I re-did the function:

	vec3 line (vec2 xy, float local_scale) {
		float f_of_x = sin(xy.x+TIME/2.);
		float abs_raw_distance = abs( xy.y - f_of_x );
		float illuminated_pixel_test = step(abs_raw_distance, 0.02);
		return vec3(illuminated_pixel_test);
	}

With the function refactored, we can then revise the color line that updates the color to this:

	col = col + line(uv, scale);

WARNING: I'm exporting the scale only because the scale might be useful for deciding line thickness, and NOT because the scale is somehow used to determine where the line exists. We're feeding raw XY coordinates into this thing all the time. As you can see, I'm not even using the local_scale parameter anywhere. But maybe, down the road, if the scale is really really big, then maybe you want the line to get thicker so it doesn't become invisible, and having that information helps you accomplish that goal.

ABSTRACTING OUT THE GRID

The grid function is already quite abstract, but we can make it consistent with the improved line function by passing in uv as a vector and scale as a float.

	col = col + grid_line(uv.x, uv.y);

Changes to...

	col = col + grid_line(uv, scale);

And the grid function, which is now this:

	vec3 grid_line (float uv_x, float uv_y) {
		//Take fractional parts of X and Y
		float x_fract = fract(uv_x);
		float y_fract = fract(uv_y);
		//Test each part to see if the fractional part is close to zero or almost one.
		float grid_test = step(x_fract, 0.02) + step(0.98, x_fract) + step(y_fract, 0.02) + step(0.98, y_fract) ;
		float lit = min(1.0, grid_test);
		vec3 grid_light = lit * vec3(0.0, 0.5, 0.2);
		return grid_light;
	}

Becomes this:

	vec3 grid_line (vec2 xy, float local_scale) {
		//Take fractional parts of X and Y
		float x_fract = fract(xy.x);
		float y_fract = fract(xy.y);
		//Test each part to see if the fractional part is close to zero or almost one.
		float grid_test = step(x_fract, 0.02) + step(0.98, x_fract) + step(y_fract, 0.02) + step(0.98, y_fract) ;
		float lit = min(1.0, grid_test);
		vec3 grid_light = lit * vec3(0.0, 0.5, 0.2);
		return grid_light;
	}

ABSTRACTING OUT THE UV MATH 

We're almost there. Here's our current UV math:

	vec2 uv = UV;
	uv = uv * 2.0 - 1.0;
	uv = uv * vec2(1.0, -1.0);
	uv = uv * scale;

To give ourselves a destination, add this line to the top of fragment():

vec2 uv = uv_math(UV, scale);

Then cut and paste that uv math to make a uv_math() function that will take the uv and the scale as arguments:

	vec2 uv_math(vec2 xy, float local_scale){
		vec2 uv = xy;
		uv = uv * 2.0 - 1.0;
		uv = uv * vec2(1.0, -1.0);
		uv = uv * local_scale;
		return uv;
	}

And now, the entire fragment() function is an extremely short read:

	void fragment() {

		vec2 uv = uv_math(UV, scale);
		vec3 col = vec3(0.);
		col = col + line(uv, scale);
		col = col + grid_line(uv, scale);
		ALBEDO = col; ALPHA = 1.0;
	}

THE REAL GRAPHING CALCULATOR WAS INSIDE US ALL ALONG 

As you can see, the line between shader and graphing calculator is very, very thin. All we've really done is taken a graphing calculator and make it behave more like one we're used to. 

AN INSIGHT INTO COMING PROBLEMS 

We can already see that we have two related problems already showing in the cracks of our graphing calculator: Scale, and aliasing. At a high scale, the grid implodes. At any scale, there is visible, ugly aliasing of both the grid and the line. How do you solve the aliasing?

Do not say "run a 4x antialiasing super sample on the whole thing." If you say that, you get sent to some kind of shader programmer hell when you die, and it's just like heaven, except it's at one fifth the framerate. We don't want that for you.

Stop the movement of time on the sin curve so you can look at its aliasing.

The existing step() function gives either a 0 or 1, which is a hard fall off. 

	vec3 line (vec2 xy, float local_scale) {
		//float f_of_x = sin(xy.x+TIME/2.);
		float f_of_x = sin(xy.x);
		float abs_raw_distance = abs( xy.y - f_of_x );
		float illuminated_pixel_test = step(abs_raw_distance, 0.02);
		//Anti-aliasing
		illuminated_pixel_test = 1.0 - smoothstep(0.015, 0.025, abs_raw_distance);
		////A wider band gives puffy or neon, depending on execution
		illuminated_pixel_test = 1.0 - smoothstep(0.005, 0.05, abs_raw_distance);
		vec3 color = vec3(0.2, 0.4, 0.8);
		return vec3(illuminated_pixel_test) * color;
	}

HOW DOES SMOOTHSTEP WORK?

I said earlier that for the step function, you could think of it as, "is the second argument higher than the first." Maybe it would be helpful to visualize the smoothstep function. Maybe it would be helpful if we had a graphing calculator that could show us what it does.

Copy and paste the line function as line2 and add its result in the fragment shader.

	col = col + line2(uv, scale);

And then have the line2 function show you what a smoothstep applied to sin would look like:


vec3 line2 (vec2 xy, float local_scale) {
	//float f_of_x = sin(xy.x+TIME/2.);
	float f_of_x = smoothstep(0.2, 0.8, sin(xy.x+TIME/2.));
	float abs_raw_distance = abs( xy.y - f_of_x );
	float illuminated_pixel_test = step(abs_raw_distance, 0.02);
	//Anti-aliasing
	illuminated_pixel_test = 1.0 - smoothstep(0.015, 0.025, abs_raw_distance);
	////A wider band gives puffy or neon, depending on execution
	illuminated_pixel_test = 1.0 - smoothstep(0.005, 0.05, abs_raw_distance);
	vec3 color = vec3(0.8, 0.4, 0.8);
	return vec3(illuminated_pixel_test) * color;
}

I've added time back in and altered the color of the second line.

WHY THE ONE MINUS BIT 

The inclusion of the one minus in this clause:

illuminated_pixel_test = 1.0 - smoothstep(0.015, 0.025, abs_raw_distance);

...feels a little weird. But that's because in the before times, our step function was testing whether 0.025 was greater than x, the distance. In the new smoothstep function, if the distance is greater than the second edge, we get a one. If it's less than the first edge, we get a zero. That's literally the opposite of what we want. "One minus the expression" fixes the problem, for when we want to test for a low number.
