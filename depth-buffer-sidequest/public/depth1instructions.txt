Now to explain each line.

    uniform sampler2D godot_depth_sampler : hint_depth_texture;

Uniforms, in theory, come from outside the shader, and stay constant within the four walls of the shader. So maybe they come from gdscript, or you set them in the inspector, or maybe they're making a trip from elsewhere in the GPU in time or space. They're a little confusing, in that when you write uniforms into your shader, you can declare them to have whatever value you want--so even though they're exposed for editing in the shader, you might forget that they're actually pulling their values from the Inspector. (Which is why you might go insane debugging a shader only to realize that you changed a value in the Inspector and that tweaking the default in the shader code does nothing.)

WAIT WHAT?

Say you're creating a uniform that's a float between 0 and 1, with a default value of 0.4. It would read like this:

    uniform float example_number : hint_range(0.0,1.0) = 0.4;

What happens if you don't give it a default, and just write this:

    uniform float example_number2 : hint_range(0.0,1.0);

Nothing bad happens. You're just now required to set the number in the Inspector if you want the number to be something other than 0, which is what it defaulted to here.

This means the word "hint" isn't really a hint. It's an order. The colon ":" is handing the job off to another party (usually the CPU) to dig up the right uniform. The equals sign "=" which you'd normally expect to function as an assignment that overrides everything else does not, in fact, override everything else.

SO WHAT IS A SAMPLER 

A sampler is "more than a texture". If you want a metaphor, I'd say that your textures are like the raw data picked up by satellites and those space-travelling telescopes that take black and white photos of space. They're able to see a lot of space-crap, some of which the human eye can see, and other things the human eye can't see. If you want to turn that machine's data into a beautiful color poster for your college dormitory wall to ponder, the people at NASA have to make choices about sampling the image. How do they colorize it? How do they brighten up the contrast? What human-invisible parts do they make visible, and what arbitrary colors do they assign? The path that data takes from being taken in by the satellite/telescope to becoming your dorm room poster is similar to the sampler to texture path. Both involve machine data taking an arbitrary path to a human-readable form. That said, you might not use it for that purpose--the sampler might be preparing the data for some other machine process, so "making it visible to people" is just one purpose, not the only purpose.

Thus, via the sampler, a depth texture isn't just a texture--it's a reflection that Godot already made a choice about how that depth texture will be processed and presented to you. This matters later.

THE LINE OF CODE YOU'D USE IF THE DepthBufferViewer COVERED THE ENTIRE SCREEN 

    //float depth = texture(godot_depth_sampler, SCREEN_UV).x;

Normally with shaders we care about the UV terrain of whatever object has the material on it. The top left of the plane's default UV is 0,0 and the bottom right's default UV coordinate is 1,1. 

SCREEN_UV is like saying, "I don't care about what the object's UV coordinate is, I actually care about the UV coordinate of this particular pixel on the monitor". Try switching back and forth between the SCREEN_UV and the UV lines to see the difference. You might have to enlarge the DepthBufferViewer to cover up more of the screen and its objects so that you can see what is, and isn't, contributing to the depth buffer.

This is useful for using a plane like a gel in front of the camera. You don't have to match up the plane's UV coordinates with the camera's UV coordinates, which can be tedious.

    float depth = texture(godot_depth_sampler, UV).x;

We are extracting a float and naming it "depth". The next clause:

texture(godot_depth_sampler, UV)

...is a function taking two arguments. It is *NOT* returning an entire texture. The first argument is the image data from the sampler, and the second argument is the XY coordinate of the sampler's image that you want to pull color data from. 

In plain English, this texture() function is what normal humans would refer to as sampling a texture at a specific position. But we're not using normal human language here. 

There's a final swizzle added:

texture(godot_depth_sampler, UV).x

...because the depth_buffer stores depth in the first channel, sometimes called the r channel (of "rgb") but here it's the x channel, of "xyz". In this particular case, the ".x" does NOT mean a position in XYZ space.

    ALBEDO = vec3(depth);

This final line should be pretty obvious if you've read my other chapters--the float for depth gets pasted into all three color channels and gets spit out as the final color, making a black and white image representing depth.

THIS DEPTH BUFFER IS HARD TO SEE

The depth buffer is expressed exponentially. This is so that you get more depth accuracy for close objects, at the expense of depth accuracy for far objects.

This is Claude's suggested code for converting that exponential version back into a linear version:

    uniform float far_distance : hint_range(0.0,100.0) = 30;

    float linear_depth(vec2 uv, mat4 inv_proj) {
        float d = texture(godot_depth_sampler, uv).x;
        vec3 ndc = vec3(uv * 2.0 - 1.0, d);
        vec4 view = inv_proj * vec4(ndc, 1.0);
        view.xyz /= view.w;
        return -view.z;
    }

...which requires updating the depth variable to instead be this:

    depth = linear_depth(UV, INV_PROJECTION_MATRIX) / far_distance;

What happens when you "multiply" two 1x2 matrices together? That's probably the first place to start, since we've done versions of that a bunch already, when we smash two vec2s together. Say we've got this:

11 + 10xy.

Let's say we have a scale(), rotate(), and translate() function that each move vertices around. Matrix multiplication allows you to combine all three functions into one function.

In graphics programming, this means this:

WHAT IS DIVISION/INVERSION?

As mentioned previously, we're not actually multiplying, when doing matrix "multiplication", so it should come as no surprise that we can't actually divide. "Division" in this case just means undoing the effects of the function. This is called inverting. So if you created the inverse of that SRT matrix, doing the multiplication/dot-producting of that inverse would restore you to your original position.

Consider our translation matrix T:
[1, 0, 0, 4]
[0, 1, 0, 5]
[0, 0, 1, 0]
[0, 0, 0, 1]

We already know that the 4,5,0 numbers in the top right are what do the translating. Which means the inverse of the translation matrix is to undo that transform. If the previous matrix translates things 4 units along X and 5 units along Y, then you move it back negative 4 units on X and negative 5 units on Y, by negating the 4 and the 5:

[1, 0, 0, -4]
[0, 1, 0, -5]
[0, 0, 1,  0]
[0, 0, 0,  1]

You may have figured out the scale matrix just from looking at it. The rotation one is the only hard one, as it requires geometry (or you just memorize or look up the formula if you need it). So creating the inversions of those isn't too crazy either. Conversely, creating the inverse of a matrix that doesn't fit into these particular patterns is difficult and requires a lot of unrelated math.

The notation of inversions is a little funny. The inverse of T is written as T^-1, just as 1/x = x^-1. The notation thus keeps the pattern of a algebraic division, but its only a philosophical similarity: It undoes the function, it's not division.
